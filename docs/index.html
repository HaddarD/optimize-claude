Context Window vs Usage Limit

What actually causes compression — and what actually drains your plan budget

🔑 One Pool of Tokens, Two Different Consequences

Every token generated — input, output, and (on current models) thinking — counts toward both your context window and your usage budget at the same time. They are not separate pools. The same token simultaneously fills your context window AND drains your usage allowance.

📐 Context Window (Compression)
✓ Filled by: your messages + my visible responses + thinking tokens + system prompts + tool definitions
✓ On Sonnet 4.6+, Opus 4.5+, and Fable 5: thinking blocks from previous turns DO stay and accumulate in context
✗ Only older models (pre-4.5 Opus, pre-4.6 Sonnet, all Haiku) strip thinking blocks between turns
⚠ Higher effort + thinking ON = more tokens generated = faster compression, not the same speed
💰 Usage Limit (Budget)
✓ Depleted by: the exact same tokens — input, output, thinking, tool calls
✓ Billing weight differs by model: Sonnet 1× · Opus ~1.67× · Fable 5 ~2×
✓ Same tokens, different price tag per model — but identical context window impact
✓ Resets on a rolling window (5-hour session + weekly cap on Pro/Max)
💡 The core insight
Switching from Opus 4.8 High (thinking ON) to Sonnet Low (thinking OFF) reduces both usage burn AND compression speed — because thinking tokens now occupy real context window space on these models. A bigger context window (e.g. 1M on Claude Code/Max, or Fable 5's 1M chat window) gives you more total room, but within that room, higher effort and thinking ON still fill it up faster per turn. Bigger window + heavy settings can still compress fast — just later than a smaller window would.

Practical tip: the two biggest levers for reducing compression are (1) keeping your own messages concise — long pastes and verbose prompts fill the window fast — and (2) turning thinking OFF and effort low for casual work, since accumulated thinking tokens now count toward context on current models. Use Projects with RAG for large documents instead of pasting them directly.
⚠️ Model Switching & Context Window Mismatch

Switching models mid-conversation is possible, but watch the context window size when you do. Going from a large-window model (Sonnet 5 or Fable 5, both 1M) to a smaller-window model (Opus 4.6/4.7/4.8 at 500K in chat) mid-chat can trigger an immediate compression if you've already used more tokens than the smaller window holds. When escalating from a long Sonnet 5 session to Opus, starting a brand new chat is usually safer and cleaner than switching in place.

Is it worth the big jump vs. grinding up the ladder?

Jump straight to Opus 4.6+ when:

• The problem clearly spans multiple files / deep reasoning

• Sonnet already failed 2–3 times at Medium or High

• It's a research/science task (GPQA gap is 17 points)

• You're already compressed and need to solve fast

Grind the ladder when:

• It's likely a simple bug or logic error (Sonnet 5 handles most of these)

• You haven't tried Thinking ON yet at the current model

• Context is mostly fresh — you have room to try a few times

• The SWE-bench gap for your task type is small

⚠ Switching models mid-chat — window mismatch risk

You CAN switch models within the same conversation. But if you switch to a model with a smaller context window than what you've already used — like Sonnet 5's 1M down to Opus 4.8's 500K — the switch can trigger immediate compression, since the conversation is already past the smaller model's ceiling. Fable 5 is the exception: same 1M window as Sonnet 5, so switching to it mid-chat carries no extra risk. Safest move when escalating from a long Sonnet 5 session: start a fresh chat with Opus 4.8 rather than switching in place.

⛔ Usage Credits Warning

Usage credits are pay-as-you-go at standard API rates — they are charged separately on top of your monthly subscription and can cost thousands of dollars compared to your $20-200/mo plan.
They are NOT required to use Claude Code. They are only required to access the 1M context window on Pro (Opus or Sonnet) and on Max (Sonnet only).
Without usage credits: Claude Code falls back to 500K context and auto-compacts or pauses at the limit — same behavior as chat. You can always /compact or /clear to continue. It does NOT permanently freeze.
The billing gate bug: Some Pro users reported Claude Code defaulting to a 1M model and hard-blocking all usage demanding credits — even on fresh sessions with quota available. If this happens: run /model and select a non-[1M] variant, or set CLAUDE_CODE_DISABLE_1M_CONTEXT=1 in your settings.

📎 Note: Why Claude Haiku 4.5 Is Not In This Calculator

Haiku 4.5 was deliberately excluded from the comparisons above. Here's what you need to know about it.

Token cost vs context window — the trap

Haiku is roughly 3x cheaper per token than Sonnet 4.6 — 1,000 Sonnet tokens ≈ 330 Haiku tokens. At Low effort with Thinking OFF, that gives you approximately 606 Haiku turns vs 500 Sonnet turns before first compression. On paper, slightly better.

But Haiku's context window is 200K tokens across all surfaces — chat, Claude Code, everything — versus Sonnet 5's 1M in chat (all paid plans) and Sonnet 4.6's 500K in chat. The token savings and the smaller window roughly cancel each other out at Low effort. The moment you raise effort or turn thinking on, the savings vanish and Haiku fills up significantly faster than Sonnet — not slower. A Haiku session at Medium or High effort will compress sooner than Sonnet at the same settings, despite costing less per token.

⚠️ Critical: Haiku does not handle complex tasks well — and the gap widens fast

Haiku 4.5 does not support extended thinking properly for complex tasks. While it technically has a thinking toggle, its reasoning depth degrades significantly on anything hard — the quality gap vs Sonnet isn't just the benchmark numbers, it actively gets worse as tasks become more demanding.

Benchmark comparison: SWE-bench Verified — Haiku 73.3% vs Sonnet 79.6% (6.3 point gap). GPQA Diamond (science/reasoning) — Haiku 55.2% vs Sonnet ~74% (nearly 20 point gap). Chatbot Arena ELO for conversation and writing quality: significantly lower. These gaps compound on harder work.

Who Haiku is actually for

API developers paying per token at high volume — teams running thousands of calls per day where cost-per-call matters more than reasoning depth. Classification, routing, tagging, boilerplate generation, simple chatbots, speed-critical applications where 80-120 tokens/second response time is the feature.

For monthly subscribers (Pro or Max), Haiku offers no meaningful advantage. You're not billed per token — you have a usage budget. Choosing Haiku to "save tokens" on a subscription doesn't reduce your bill. It just gives you a less capable model with a smaller context window that compresses faster at any real effort level. Not recommended for personal subscription use.

Quick reference: Haiku 4.5 — $1/$5 per million tokens (API) · 200K context (all surfaces) · 73.3% SWE-bench · 55.2% GPQA Diamond · 80-120 tok/sec · No effective extended thinking on complex tasks · Best for: API volume, speed-critical apps · Not for: monthly subscribers, complex reasoning, emotional writing, long-context work
